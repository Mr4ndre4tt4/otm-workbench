# Load Plan CSVUTIL Builder Design

## Context

Load Plan can now register approved Rates CSV exports as backend-only
`LoadPlanPackage` records. Each package links to the upstream Rates ZIP artifact,
manifest, approval evidence, and package intake evidence.

The next Load Plan roadmap item is CSVUTIL/CTL/CL generation. This slice creates
that builder contract without executing anything against OTM, without ZIP
analysis, without setup review, and without cutover readiness.

## Goal

Add backend/API support for generating internal CSVUTIL control artifacts from a
registered Load Plan package, preserving artifact, manifest, evidence, audit,
and domain-event metadata.

## Scope

Included:

- New `CsvutilBuild` persistence model.
- Alembic migration for `csvutil_builds`.
- `POST /api/v1/modules/load-plan/csvutil/build`.
- `GET /api/v1/modules/load-plan/csvutil/builds`.
- `GET /api/v1/modules/load-plan/csvutil/builds/{build_id}`.
- Generation of a CTL artifact (`artifact_type="csvutil_ctl"`).
- Generation of a CL artifact (`artifact_type="csvutil_cl"`).
- Generation of a manifest (`manifest_type="csvutil_build"`).
- Client-safe evidence (`evidence_type="csvutil_build"`).
- Audit log action `load_plan.csvutil.build`.
- Domain event `load_plan.csvutil.built`.

Excluded:

- Running CSVUTIL.
- Connecting to OTM.
- Validating whether OTM accepted the package.
- ZIP analysis.
- Setup review queue.
- Review decisions.
- Cutover readiness.
- Cutover package export.
- UI.

## Data Dictionary And CSV Rules

The builder must use the `LoadPlanPackage.load_sequence_json` produced from
Rates table metadata. It must not infer load order from ZIP file order or from
ad hoc filename sorting.

The source package already points to a Rates export ZIP generated by the
Data Dictionary-backed CSV builder. If date-bearing OTM tables are included in
that upstream ZIP, the CSV content must already follow the existing rule:

```text
table name line
column header line
exec alter session set nls_date_format = 'YYYY-MM-DD HH24:MI:SS'
value lines
```

This slice does not rewrite OTM CSV files. It creates control artifacts that
refer to the package files and table sequence.

## Model Contract

Add `CsvutilBuild`:

```text
id
project_id
environment_id
profile_id
package_id
status
ctl_artifact_id
cl_artifact_id
manifest_id
evidence_id
summary_json
created_by
built_at
created_at
updated_at
```

Initial status:

```text
BUILT
```

This status means the local control artifacts were generated. It does not mean
CSVUTIL ran successfully, OTM accepted the files, or the package is ready for
cutover.

`summary_json` example:

```json
{
  "package_id": "load_package_id",
  "source_module": "rates",
  "source_entity_type": "rate_batch",
  "source_entity_id": "batch_id",
  "package_type": "rates_csv_zip",
  "table_count": 1,
  "row_count": 1,
  "ctl_artifact_type": "csvutil_ctl",
  "cl_artifact_type": "csvutil_cl"
}
```

## Artifact Contract

Generate files under:

```text
var/artifacts/load_plan/{package_id}/csvutil/{timestamp}/
```

Files:

```text
csvutil_{package_id}.ctl
csvutil_{package_id}.cl
manifest.json
```

Artifact rows:

```text
artifact_type = "csvutil_ctl"
content_type = "text/plain"
sensitivity_level = "internal"
```

```text
artifact_type = "csvutil_cl"
content_type = "text/plain"
sensitivity_level = "internal"
```

The CTL/CL artifacts are internal because they contain operational file names and
load order, but they must not contain raw CSV row values.

### CTL Content

The MVP0 CTL is a deterministic, line-oriented control preview:

```text
# OTM Workbench CSVUTIL CTL
# package_id={package_id}
# source_module={source_module}
001,ACCESSORIAL_COST,csv/001_ACCESSORIAL_COST.csv
```

The table rows must follow `LoadPlanPackage.load_sequence_json` order. The CSV
relative filename should match the Rates export convention:

```text
csv/{sequence:03d}_{table_name}.csv
```

### CL Content

The MVP0 CL is a deterministic command-list preview:

```text
# OTM Workbench CSVUTIL CL
# package_id={package_id}
LOAD ACCESSORIAL_COST FROM csv/001_ACCESSORIAL_COST.csv
```

This is a builder artifact for review and future integration. It is not a claim
that an Oracle CSVUTIL runtime will accept the command unchanged.

```

  "source_module": "load_plan",
  "source_entity_type": "load_plan_package",
  "source_entity_id": "load_package_id",
  "package": {
    "id": "load_package_id",
    "source_module": "rates",
    "source_entity_id": "batch_id",
    "package_type": "rates_csv_zip"
  },
  "files": [
    {
      "artifact_type": "csvutil_ctl",
      "file_name": "csvutil_load_package_id.ctl",
      "sha256": "hash",
      "size_bytes": 123
    },
    {
      "artifact_type": "csvutil_cl",
      "file_name": "csvutil_load_package_id.cl",
      "sha256": "hash",
      "size_bytes": 123
    }
  ],
  "load_sequence": [
    {
      "position": 6,
      "table_name": "ACCESSORIAL_COST",

  "built_by": "admin@example.com"
}
```

The manifest must not include raw CSV rows, charge amounts, lane payloads, or
real client identifiers.

## Evidence Contract

Create one `Evidence` row per build:

```text
source_module = "load_plan"
evidence_type = "csvutil_build"
status = CREATED
client_safe = true
sensitivity_level = client_safe
manifest_id = csvutil build manifest id
artifact_id = ctl artifact id
```

`summary_json`:

```json
{
  "source_entity_type": "csvutil_build",
  "source_entity_id": "build_id",
  "package_id": "load_package_id",
  "source_package_type": "rates_csv_zip",
  "table_count": 1,
  "row_count": 1,
  "ctl_artifact_id": "artifact_id",
  "cl_artifact_id": "artifact_id",
  "manifest_id": "manifest_id",
  "status": "BUILT"
}
```

Evidence should summarize the generated control package only. It must not copy
CTL/CL content, ZIP contents, CSV contents, or manifest file details.

## API Contract

### Build CSVUTIL Artifacts

```text
POST /api/v1/modules/load-plan/csvutil/build
```

Request:

```json
{
  "package_id": "load_package_id"
}
```

Response:

```json
{
  "id": "csvutil_build_id",
  "package_id": "load_package_id",
  "status": "BUILT",
  "ctl_artifact_id": "artifact_id",
  "cl_artifact_id": "artifact_id",
  "manifest_id": "manifest_id",
  "evidence_id": "evidence_id",
  "summary": {
    "table_count": 1,
    "row_count": 1,
    "package_type": "rates_csv_zip"
  }
}
```

### List Builds

```text
GET /api/v1/modules/load-plan/csvutil/builds
```

Use existing `PageResponse`.

### Get Build

```text
GET /api/v1/modules/load-plan/csvutil/builds/{build_id}
```

Returns the build plus parsed `summary`.

## Preconditions

Build is allowed when:

- package exists;
- package status is `REGISTERED`;
- package has `artifact_id`;
- package has `manifest_id`;
- package `load_sequence_json` is non-empty;
- package load sequence has at least one row across tables.

Reject when:

- package does not exist: 404;
- package is not `REGISTERED`: 400;
- package has no source artifact/manifest: 400;
- load sequence is empty: 400.

Build should not be idempotent in this slice. Rebuilding is allowed and creates a
new `CsvutilBuild` with fresh artifacts. This preserves a local history of
generated control files as package or builder rules evolve.

## Capability Note

The broader architecture names `load_plan.csvutil.generate`, but the current
backend only has shared authenticated-user dependencies. This slice should
require authentication and record `created_by`. Full capability enforcement can
be added when module-level capability dependencies exist.

## Testing Strategy

Minimum tests:

- build rejects missing package;
- build rejects package without source artifact/manifest;
- build rejects package with empty load sequence;
- build succeeds for registered Rates package;
- CTL artifact is created with deterministic table order;
- CL artifact is created with deterministic table order;
- manifest references both artifacts and package metadata;
- evidence is client-safe and excludes raw row values;
- audit log and domain event are created;
- list builds returns generated builds;
- get build returns parsed summary;
- repeated builds create separate `CsvutilBuild` rows;
- Alembic upgrade creates `csvutil_builds`.

## Risks

CSVUTIL artifacts can be mistaken for execution. Mitigation: use status `BUILT`
only, never `EXECUTED`, and explicitly avoid OTM connection or runtime calls.

CTL/CL syntax can drift from the eventual Oracle runtime command format.
Mitigation: label the generated files as deterministic MVP0 builder artifacts
and keep execution out of scope until runtime requirements are confirmed.

Evidence can leak operational details. Mitigation: evidence stores IDs, counts,
hash-linked artifacts, and status only.

## Decisions

- Rebuilds are allowed and produce new build history.
- CTL and CL are separate artifacts rather than a ZIP in this slice.
- Manifest stores hashes and load sequence, not raw file content.
- Source CSV ZIP remains the upstream Rates artifact; this builder does not
  rewrite or unzip CSV files.
- No `LoadPlan` execution entity is introduced yet.
