# Evidence Hub Index Design

## Context

The backend now creates platform `Evidence`, `Artifact`, and `Manifest` records
across Rates and Load Plan slices. Each module exposes only narrow module-local
lookups, so there is still no canonical Evidence Hub surface for browsing
evidence generated by different workflows.

The functional architecture defines Evidence Hub as the central place for
evidence, artifacts, manifests, status, and traceability. This slice implements
the first backend-only index for that surface.

No OTM table dependency or CSVUTIL behavior is introduced in this slice. The
local OTM Data Dictionary remains the source for module-specific OTM dependency
logic, but Evidence Hub only reads already-persisted platform metadata.

## Decision

Implement **Evidence Hub Index** first, before artifact download or archive
packages.

Compared options:

- **Download first:** useful, but it exposes filesystem streaming and permission
  behavior before users have a unified way to discover evidence.
- **Archive package first:** valuable later, but it requires deciding project
  phases and archive contents before the index contract exists.
- **Index first:** lower risk, backend-only, and gives every module a canonical
  read surface for evidence traceability.

Recommendation: implement index first.

## Goals

- Add a backend/API Evidence Hub router.
- List evidence records across modules with filters.
- Get one evidence detail by id.
- Include linked artifact and manifest summaries when available.
- Keep responses client-safe by default.
- Avoid exposing raw artifact files or full manifest payloads in this slice.

## Non-Goals

- No artifact download endpoint.
- No archive package generation.
- No new evidence-writing workflow.
- No OTM upload, CSVUTIL execution, external Oracle calls, or package status
  transition.
- No UI.
- No real client names or real customer data in docs/tests.

## API

Base path:

```text
/api/v1/evidence-hub
```

List evidence:

```text
GET /api/v1/evidence-hub/evidence
```

Filters:

```text
source_module
evidence_type
status
project_id
client_safe
sensitivity_level
artifact_id
manifest_id
```

Sort:

- newest first by `created_at`.


The detail response uses the same item shape as list entries.

## Data Contract

Evidence summary:

- Parse `Evidence.summary_json` into `summary`.
- If parsing fails or the stored payload is not an object, return `{}` instead
  of leaking raw text.

Artifact summary:

- Include artifact metadata only.
- Do not include `file_path`.
- Do not stream or download content.

Manifest summary:

- Include manifest id, source module, status, created timestamp.
- Parse `Manifest.manifest_json` only to extract top-level `manifest_type` and
  `schema_version`.
- Do not return full `manifest_json` in this slice.

## Security And Permissions

Use the existing authenticated route dependency for MVP0 consistency. Later
hardening can add capability checks for Evidence Hub viewing and artifact
download.

Client-safe default:

- `GET /evidence` returns only `Evidence.client_safe == true` unless the caller
  explicitly sends `client_safe=false`.
- MVP0 admin/user distinction is unchanged; both still require authentication.
- No raw CSV row values, review decision notes, file paths, or real client names
  should appear in Evidence Hub responses.

## Data Safety

- Tests use synthetic identifiers such as `OTM1`, `DEMO`, and generated ids.
- Evidence Hub must not reconstruct module-specific payloads.
- Evidence Hub must not inspect artifact ZIP contents.
- Evidence Hub must not expose raw `Manifest.manifest_json`.
- Evidence Hub must not expose `Artifact.file_path`.

## Tests

Add backend tests with synthetic data:

- Evidence Hub list requires authentication.
- List returns client-safe evidence by default.
- List supports filters by module, type, status, project, sensitivity, artifact,
  and manifest.
- Detail returns artifact and manifest summaries.
- Detail rejects missing evidence with 404.
- Response does not include artifact `file_path`.
- Response does not include full `manifest_json`.
- Response does not leak raw CSV values or review notes from linked module
  workflows.

## Risks

Users can mistake Evidence Hub for an artifact download surface. Mitigation:
this slice names linked artifacts but does not expose file paths or downloads.

Manifest payloads can contain technical details not meant for the client-safe
index. Mitigation: expose only `manifest_type` and `schema_version`.

Evidence records from older workflows may have inconsistent summaries.
Mitigation: parse defensively and return empty objects for invalid summaries.

## Next Steps

After this slice, the next likely increments are:

1. Evidence Hub artifact download with explicit permission/audit behavior.
2. Evidence Hub archive package by project or operational phase.
3. Cutover package eligibility/status transition once download/archive behavior
   is settled.
