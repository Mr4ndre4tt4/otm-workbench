# Workbench Assistant Governance And Risks

## Purpose

The Workbench Assistant crosses several sensitive boundaries: private client
data, environment scope, Data Dictionary truth, Oracle documentation freshness,
SQL generation, and optional AI/web costs. This document defines governance
rules and risks for future implementation.

## Governance Rules

### G1: Source-Bound Answers

Important answers must cite sources. If the assistant cannot identify a source,
it should say that it cannot confirm the answer.

Applies to:

- Oracle technical answers;
- SQL drafts;
- Data Dictionary claims;
- file/template/evidence results;
- module usage instructions.

### G2: Scope Before Snippet

Permission and scope checks must happen before titles, snippets, paths, or
content are returned.

The assistant must not leak private source metadata through "almost found"
messages.

### G3: Public View Is Separate

Public View can use:

- public Workbench help;
- public/shared templates only when explicitly marked public;
- official Oracle documentation;
- public Data Dictionary information.

Public View cannot use private client/domain files, artifacts, evidence, or
saved queries.

### G4: SQL Is Draft-Only

SQL generated by the assistant is a draft. The assistant should not execute SQL
against OTM or any customer system.

The first design supports select-only drafts. Mutating SQL should be rejected
unless a future DBA-only workflow is explicitly approved.

### G5: Oracle Docs Require Official Source

Oracle technical answers that depend on external documentation must include an
official Oracle link or clearly state that no official source was found.

### G6: Optional AI Cannot Bypass Tools

If an AI adapter is added later, it can summarize or rewrite source-grounded
answers. It cannot:

- bypass scope filtering;
- invent source citations;
- invent Data Dictionary tables/columns;
- answer Oracle docs questions without official sources;
- create SQL outside SQL helper rules.

### G7: Costs Must Be Visible

Cost-bearing operations must be represented in the response contract:

```text
local
web
ai
web_plus_ai
```

Live Oracle lookup and optional AI should be explicit modes or actions.

## Risk Register

| ID | Risk | Impact | Mitigation | Status |
|---|---|---|---|---|
| WA-R001 | Assistant becomes a generic chatbot. | Low trust, vague answers, scope creep. | Tool-based intents, source-bound responses, no "ask anything" framing. | Open |
| WA-R002 | Private client data leaks through search snippets. | Confidentiality failure. | Scope-before-snippet rule and denied-result non-leakage tests. | Open |
| WA-R003 | SQL helper invents table/column relationships. | Wrong implementation decisions. | Data Dictionary grounding, approved join patterns, clarification on ambiguity. | Open |
| WA-R004 | Oracle docs answers become stale. | Incorrect technical guidance. | Live official lookup, cache timestamps, expiry, refresh action. | Open |
| WA-R005 | Online/API costs grow invisibly. | Unexpected operating cost. | Cost-level metadata, explicit web mode, caching. | Open |
| WA-R006 | Local indexing becomes too slow. | Poor usability on consultant laptops. | Incremental indexing, source hashes, health report, no heavy models. | Open |
| WA-R007 | Assistant UI covers important module controls. | Workflow friction. | Lower-right closed state, right panel open state, responsive QA. | Open |
| WA-R008 | Public View accidentally shows private source existence. | Scope trust failure. | Generic denied copy; do not reveal titles, paths, or counts. | Open |
| WA-R009 | Optional AI is mistaken as the source of truth. | Hallucinated or over-trusted answers. | AI disabled by default; source/citation layer remains authoritative. | Open |
| WA-R010 | Assistant planning distracts from active module delivery. | Roadmap drift. | Keep as separate planning track until explicitly promoted. | Open |

## Decision Log Seeds

These decisions should be logged if the assistant is promoted into active
roadmap scope:

1. Workbench Assistant is a cross-cutting embedded utility, not a top-level
   module.
2. Local-first structured search is the foundation; heavy local LLM is not.
3. SQL helper is select-only and draft-only in the initial design.
4. Oracle docs lookup is controlled, official-source-only, and cache-aware.
5. Optional AI is an enhancement layer, not a dependency.

## Review Gates

Before implementation:

- planning packet approved;
- SQL helper safety accepted;
- source index scope rules accepted;
- UI entry reviewed by Michelangelo/Figma;
- dependency additions reviewed for local footprint.

Before integration into main shell:

- source index works;
- scope guard tests pass;
- launcher/panel accessibility tests pass;
- no conflict with active module navigation;
- browser QA confirms no primary controls are covered.
