# Risk Register

**Status:** active
**Date:** 2026-05-26

| ID | Risk | Impact | Mitigation | Status |
|---|---|---|---|---|
| R-001 | Stale docs remain mixed with active docs. | Agents may follow obsolete direction. | Maintain `DOCUMENT_INVENTORY.md`, classify docs, archive only after approval. | Open |
| R-002 | Cleanup starts by deleting code or docs. | Loss of evidence and hard-to-recover context. | Use reversible archive plan first; source cleanup needs separate implementation plan. | Open |
| R-003 | Wireframes copy current UI instead of validated scope. | Redesign preserves old confusion. | Use module scope ledger and Michelangelo briefs as source. | Open |
| R-004 | Module specs disagree with original foundation scope. | Implementation may drift from product purpose. | Resolve through decision log before wireframes. | Open |
| R-005 | Backend-owned truth leaks back into frontend-local maps. | Permissions, labels, actions, and lifecycle can diverge. | Require backend-owned contracts in module acceptance criteria. | Open |
| R-006 | Real client data appears in docs, screenshots, fixtures, or tests. | Security and confidentiality failure. | Keep synthetic-only rule in AGENTS, QA, and docs. | Open |
| R-007 | OTM behavior is assumed without validation. | Generated artifacts or validation rules may be wrong. | Validate through Data Dictionary and official Oracle documentation. | Open |
| R-008 | Penpot phase expands into visual polish before task clarity. | Wireframes become aesthetic instead of operational. | Michelangelo review must identify task, archetype, states, and acceptance criteria. | Open |
| R-009 | `OTM_RESOURCES/` is modified accidentally. | Local validation resources may be corrupted. | Treat as protected unless explicitly requested. | Open |
| R-010 | Linear/GitHub visibility falls behind docs. | Delivery state becomes hard to trust. | Update Linear/GitHub after validated product behavior changes. | Open |
| R-011 | Master Data wireframes copy the delivered UI too closely. | The target design may preserve old compromises instead of the validated route-family model. | Use the scope review and Michelangelo brief as source before Figma/mockup work. | Open |
| R-012 | Catalog Core wireframes expose too much technical detail to normal users. | Catalog may become a confusing technical wall instead of a focused source-of-truth explorer. | Keep the hub compact, separate route-level inspection, and show readiness/confidence labels. | Open |
| R-013 | Schema guidance is mistaken for Data Dictionary CSV/load-order truth. | Users may make incorrect implementation decisions from XML paths. | Wireframes must separate Data Dictionary truth from contract guidance and show confidence/source status. | Open |
| R-014 | Admin and Developer Tools leak restricted controls into normal workflows. | Security and role-boundary confusion. | Keep Admin and Developer Tools route families permission-gated and distinct in wireframes. | Open |
| R-015 | Evidence Hub and Assets Library become generic file managers. | Evidence, artifact, version, and audit trust boundaries weaken. | Preserve client-safe evidence, guarded downloads, backend-owned links, and archive reviews in briefs. | Open |
| R-016 | Cutover handoff appears as a dashboard action instead of governed operation. | Unready packages may be treated as ready. | Wireframe handoff as a route-level critical review with backend eligibility. | Open |
| R-017 | Login/logout are treated as incidental browser states instead of designed shell states. | Users may see stale authenticated navigation after logout or fail to recover from invalid sessions. | Include login, invalid-login, logout-confirmed/session-ended, and signed-out recovery frames in Shell wireframes. | Open |
| R-018 | Cockpit overreaches into project-management control. | The product becomes heavier than the user's accelerator model and constrains when modules can be used. | Replace Cockpit v1 readiness/workstream/blocker boards with v2 context selector, accelerator launcher, Public View, and Project Info Hub. | Open |
| R-019 | Client/domain and environment isolation is not universal. | Records may leak across projects, environments, or public/private scopes. | Treat `client/domain + environment + visibility/access policy` as a persistence invariant for all operational records. | Open |
| R-020 | Project Info Hub exposes secrets unsafely. | Credentials may leak through UI, logs, screenshots, fixtures, or weak storage. | Wireframe only secret metadata and secure reveal placeholders; require separate security design for encrypted storage, reauthentication, permissions, and audit. | Open |
| R-021 | Current Figma UI phase hides broader module dependencies. | Excluded modules could reappear accidentally as top-level routes or be forgotten as backend/internal dependencies. | Keep Catalog Core/Data Dictionary as internal validation dependency, keep Evidence module-local for now, absorb setup into Settings, and require a decision log update before reintroducing excluded UI modules. | Open |
| R-022 | The FigJam as-is diagnostics are mistaken for cleanup approval. | Routes, components, docs, or services could be removed before the user approves a cleanup candidate list. | Treat the FigJam board as diagnostic only; cleanup still requires route/component inventory, candidate classification, user approval, implementation plan, and tests. | Open |
| R-023 | Backend-rich capabilities are cut because their top-level UI is out of scope. | Internal validation, evidence, Data Dictionary, jobs, or technical safeguards could be lost. | Distinguish top-level UI removal from backend/service retention. Keep Catalog, evidence, jobs, and restricted diagnostics where they support active modules or governance. | Open |
| R-024 | Active visual source confusion between two Figma files. | Agents may update or implement from the older consolidated wireframe file instead of the Complete Solution Mockup. | Treat the Complete Solution Mockup as the active To-Be source and keep the consolidated wireframe file as supporting evidence only. | Open |
| R-025 | Frontend cleanup removes routes before tests and dependencies are classified. | Useful backend/internal dependencies or evidence routes may be broken unintentionally. | Start with inventory and hiding/pruning of navigation exposure; delete only after explicit approval, implementation plan, and tests. | Open |
| R-026 | Data segregation is implemented partially. | Records may still leak across client/domain, environment, or Public View boundaries. | Make client/domain, environment, and access-policy scope a persistence/API/UI/test invariant before finishing Settings and Cockpit. | Open |
| R-027 | Synthetic fixture files are too small or unrealistic. | Tests may pass while real CSV/XML/package/document workflows fail. | Create valid module-specific fixtures, including large files where needed; validate OTM behavior through Data Dictionary and official Oracle documentation. | Open |
| R-028 | Module delivery proceeds without revalidating desired outcome. | A module may be technically functional but not solve the real user workflow. | Require module revalidation against Figma board, scope review, wireframe brief, GUI spec, tests, fixtures, and completion criteria before each delivery. | Open |
| R-029 | Accepted modules retain backlog items that are mistaken for completed scope. | Future agents may assume Rates advanced search, row mutation, and full out-of-order QA are already done. | Keep module acceptance checklists explicit about partial criteria and backlog before moving modules. | Open |
| R-030 | Browser QA runs against a stale local backend or frontend process. | Screenshots may show excluded modules or old contracts even when source tests pass, causing false acceptance evidence and scope confusion. | Before screenshots, run the runtime freshness gate: verify source navigation tests, query live `/api/v1/platform/navigation` on the browser QA backend, compare module IDs with current scope, and restart or move to a fresh port when they differ. | Open |
| R-031 | New chats lose or compress critical context from the previous session. | Agents may continue from partial memory, miss corrections, overwrite user changes, trust stale evidence, or skip required validation. | Require the chat-continuity workflow: previous chats leave a handoff capsule, and new chats rebuild state from AGENTS, HANDOFF, VALIDATION_REPORT, DECISION_LOG, RISK_REGISTER, CURRENT_SCOPE, DELIVERY_PIPELINE, git status, and relevant diffs before acting. | Open |
