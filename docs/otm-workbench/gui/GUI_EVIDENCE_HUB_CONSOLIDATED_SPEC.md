## 1. Product Objective

Evidence Hub should be the backend-first audit and handoff cockpit for
client-safe evidence, generated artifacts, manifests, downloads, archives, and
implementation proof across OTM Workbench modules.

The module exists to help an implementation user answer:

```text
1. What was generated?
2. Which module generated it?
3. Which source object, batch, checklist, mapping, or job produced it?
4. What artifact or manifest is attached?
5. Is it client-safe?
6. Can I download it?
7. Has it been archived for handoff?
8. What is the audit trail around validation, export, approval, archive, or
   download?
```

Evidence Hub is not a generic file manager. It is the cross-module proof layer
for validation, generated artifacts, readiness, approvals, exports, QA, and
handoff.

## 2. Core Concepts

Evidence Hub must keep these concepts distinct:

```text
Asset
  A reusable governed file, maintained by Assets Library.

Artifact
  A technical output generated by a job, preview, validation, package, export,
  or generator.

Evidence
  A client-safe proof record that something was validated, generated, reviewed,
  approved, archived, or decided.

Manifest
  A backend-owned metadata contract describing generated content, checksums,
  module context, source object, and safety flags.

Archive package
  A curated handoff package of evidence/artifact references generated through
  backend rules.
```

The UI should never blur these into one generic "file" concept.

## 3. Explicit Non-Goals

Evidence Hub is not, for MVP0/MVP1:

```text
- a raw artifact browser;
- a raw manifest JSON viewer;
- a local filesystem explorer;
- a production payload viewer;
- an Assets Library replacement;
- a frontend-owned permission engine;
- an unguarded bulk download page;
- a place to store client-specific payload values.
```

The backend remains the source of truth for evidence eligibility, artifact
download, archive creation, manifest safety, permissions, hash checks, audit
logging, and lifecycle state.

## 4. Current UI Review

Browser review of `/evidence` found a useful and safe first functional slice.

Delivered strengths:

```text
- evidence index is backend-backed;
- metrics show evidence, artifacts, manifests, and client-safe counters;
- filter controls exist;
- evidence detail is loaded from backend ids;
- artifact download is guarded and disabled when no artifact exists;
- archive creation exists;
- archive history is backend-owned;
- no raw payload, local file path, or raw manifest JSON is rendered;
- list density is capped with visible "showing 12 of N" behavior.
```

Current product and UX problems:

1. `/evidence` combines global evidence search, selected evidence detail,
   download action, archive action, and archive history in one route.
2. The selected-object side panel is doing too much. It should be a summary, not
   the only evidence detail screen.
3. Evidence rows do not open dedicated detail routes, so the user cannot share,
   revisit, or inspect one evidence record as a first-class object.
4. Archive package history appears inside every selected evidence context, even
   when it is global history. This makes it feel attached to the selected row
   even when it is not.
5. Filters are text inputs. This is technically flexible, but not enough for an
   audit cockpit with modules, types, statuses, sensitivity, project, artifact,
   manifest, dates, and source object context.
6. The workflow strip (`Find`, `Inspect`, `Download`, `Archive`) is accurate but
   not actionable. It does not separate screens or explain readiness.
7. Download and archive are important/audited operations, but the current UI
   treats them as side-panel buttons rather than dedicated review steps.
8. There is no clear route for archive package detail, archive contents, or
   archive audit review.
9. There is no focused view for "evidence by source object", for example all
   evidence generated by one Rates batch, Load Plan package, Integration
   definition, or Order Release batch.
10. The module is safe, but not yet strong enough as the project's audit and
    handoff control room.

Product conclusion:

```text
The current Evidence Hub is a solid functional slice and safety proof. The next
UX step is to turn it into a route-level audit cockpit with dedicated evidence
detail, artifact download review, archive builder, archive detail, source-object
timeline, and audit trail screens.
```

## 5. Design Decision

Replace the single list/detail/side-panel route with a route-level evidence
workspace.

The new IA should be:

```text
/evidence
  Evidence hub and searchable index

/evidence/items/:evidenceId
  Evidence detail

/evidence/items/:evidenceId/download
  Guarded artifact download review

/evidence/items/:evidenceId/archive
  Archive selected evidence

/evidence/archives
  Archive package library

/evidence/archives/new
  Archive package builder

/evidence/archives/:archiveId
  Archive package detail

/evidence/source/:sourceModule/:sourceEntityType/:sourceEntityId
  Source object evidence timeline

/evidence/artifacts/:artifactId
  Artifact metadata detail

/evidence/audit
  Audit trail search, if backend exposes a focused audit story
```

Every detail, download, archive, artifact, source timeline, and audit route must
have a visible `Back` action.

## 6. Global UX Rules

1. The hub is for search and triage, not deep inspection.
2. Evidence detail must be a first-class route.
3. Archive history must have its own route and must not look like selected-row
   metadata unless scoped to that evidence.
4. Download must be a guarded review action with backend-provided eligibility,
   hash/checksum metadata, sensitivity, audit intent, and disabled reason.
5. Archive creation must be a builder/review flow, not a generic side-panel
   button.
6. Evidence, artifact, manifest, archive, and audit objects must remain
   visually distinct.
7. Raw payloads, raw manifest JSON, local artifact paths, credentials, and real
   client values must never render.
8. All permissions, lifecycle state, download eligibility, archive eligibility,
   and safety decisions are backend-owned.

## 7. Screen: `/evidence`

### Purpose

Help the user find evidence across modules and decide what to inspect next.

### Backend Loads

```text
GET /api/v1/evidence-hub/evidence?filters...
GET /api/v1/evidence-hub/summary
GET /api/v1/evidence-hub/filter-options
GET /api/v1/platform/navigation
```

### Main Content

```text
- searchable evidence table;
- saved views: All, Needs review, With artifact, Archive ready, Recent exports,
  Cutover handoff, By current project;
- summary metrics: total evidence, artifacts, manifests, archives, blocked
  downloads, pending archives, client-safe ratio;
- primary action: Create archive package;
- secondary action: Archive history.
```

### Searchable Fields

Each searchable field should support:

```text
begins with
contains
one of
not one of
```

Comma-separated values are used for `one of` and `not one of`.

Recommended searchable fields:

```text
evidence_id
evidence_type
status
source_module
source_entity_type
source_entity_id
project_id
profile_id
environment_id
sensitivity
client_safe
artifact_id
artifact_file_name
artifact_type
manifest_id
manifest_schema
archive_package_id
created_by
created_at
tags
summary
```

Date filters should support explicit ranges.

### Clicks

| Click | Opens |
|---|---|
| evidence row | `/evidence/items/:evidenceId` |
| artifact link | `/evidence/artifacts/:artifactId` |
| source object link | `/evidence/source/:sourceModule/:sourceEntityType/:sourceEntityId` |
| `Create archive package` | `/evidence/archives/new` |
| `Archive history` | `/evidence/archives` |
| saved view/filter | updates backend query and URL state |

### What Must Not Be Here

```text
- raw manifest JSON;
- raw artifact content;
- local artifact paths;
- archive package contents expanded inline;
- audit log dump;
- side-panel-only primary actions.
```

## 9. Screen: Guarded Download Review

### Route

```text
/evidence/items/:evidenceId/download
```

### Purpose

Review whether an artifact can be downloaded and execute the guarded backend
download.

### Backend Loads/Actions

```text
GET /api/v1/evidence-hub/evidence/{evidence_id}
GET /api/v1/evidence-hub/evidence/{evidence_id}/download-readiness
GET /api/v1/evidence-hub/artifacts/{artifact_id}/download
```

### Main Content

```text
- artifact file name, type, size, checksum/hash;
- sensitivity;
- client-safe status;
- permission status;
- hash validation status;
- audit intent;
- disabled reason if blocked;
- download action.
```

### Actions

| Action | Backend execution | Result |
|---|---|---|
| `Download` | guarded backend artifact download | browser download and audit event |
| `Cancel` | none | back to evidence detail |

## 10. Screen: Archive Builder

```text

```

### Purpose

Build a curated package for handoff, QA, CRP, cutover, or review.

### Backend Loads

```text

```

### Main Content

```text

- selected evidence basket;
- artifact/manifest inclusion summary;

```

### Actions

| Action | Backend execution | Result |
|---|---|---|
| `Add evidence` | local selection backed by evidence ids | selected basket updates |
| `Remove evidence` | local selection | selected basket updates |
| `Validate archive` | backend archive readiness | blockers/warnings update |
| `Create archive package` | `POST /archive-packages` | navigate to archive detail |
| `Cancel` | none | back to hub or archive library |

## 15. Optional Screen: Audit Search

### Route

```text
/evidence/audit
```

### Purpose

Expose audit search only when the backend provides a focused Evidence Hub audit
contract.

This should not be a raw audit-log dump. It should answer:

```text
- who downloaded this artifact?
- who created this archive?
- what source object generated this evidence?
- did hash validation pass?
- which permission/capability allowed the action?
```

## 16. Backend Contract Gaps

The current backend supports the first slice. Route-level UX needs explicit
contracts or aliases for:

```text
GET /evidence-hub/summary
GET /evidence-hub/filter-options
GET /evidence-hub/evidence/{id}/available-actions
GET /evidence-hub/evidence/{id}/related
GET /evidence-hub/evidence/{id}/audit-summary
GET /evidence-hub/evidence/{id}/download-readiness
GET /evidence-hub/archive-packages
GET /evidence-hub/archive-packages/{id}
GET /evidence-hub/archive-packages/{id}/contents
GET /evidence-hub/archive-packages/{id}/audit-summary
GET /evidence-hub/archive-filter-options
GET /evidence-hub/source/{module}/{entity_type}/{entity_id}
GET /evidence-hub/artifacts/{id}
GET /evidence-hub/artifacts/{id}/download-readiness
```

Where these already exist under different names, the GUI should consume backend
contracts instead of inferring permissions, safety, or archive eligibility.

## 17. QA Journeys

Required browser/UI QA:

```text
1. search evidence by source module, evidence type, status, project, sensitivity,
   artifact, manifest, and date range;
2. open evidence detail and return to filtered hub;
3. open evidence with no artifact and confirm download is blocked;
4. open evidence with artifact and run guarded download review;
5. create archive from one evidence record;
6. create archive from multiple filtered evidence records;
7. open archive detail and inspect contents;
8. open source object timeline for a Rates batch;
9. open source object timeline for a Load Plan package;
10. open artifact detail and confirm no local path/raw content is shown;
11. reset filters and verify selected evidence/download/archive state clears;
12. switch selected evidence after archive creation and confirm stale feedback is
    gone;
13. verify light/dark/system mode and compact density;
14. verify no raw payload, raw manifest JSON, local path, credential, CNPJ, CPF,
    or production identifier appears.
```

Visual QA:

```text
- desktop and mobile route fit;
- high-volume evidence list;
- long artifact names and UUIDs;
- empty/no-result states;
- blocked download state;
- archive builder basket;
- archive detail contents;
- source object timeline density;
- audit summary readability.
```

## 18. Implementation Slices

Recommended implementation order:

```text
1. Hub with stronger filters and row navigation.
2. Evidence detail route.
3. Guarded download review route.
4. Archive library and archive detail routes.
5. Archive builder route.
6. Source object evidence timeline route.
7. Artifact detail route.
8. Evidence audit search only after backend contract exists.
9. Browser QA for download, archive, source timeline, filter reset, and
   route-return recovery.
```

## 19. Acceptance Criteria

Evidence Hub can be considered product-ready when:

```text
- a user can find evidence across modules without guessing exact text values;
- evidence detail is a first-class route;
- artifact download has a guarded review screen;
- archive packages are first-class objects with contents and audit context;
- source object timelines explain the proof trail for a batch/package/mapping/job;
- raw payloads, raw manifests, local paths, credentials, and client identifiers
  never render;
- permissions, safety, hash checks, archive eligibility, and audit are
  backend-owned;
- docs, tests, Linear, and GitHub point to this spec.
```

## 20. Decision Summary

The current Evidence Hub should remain as functional evidence, not final UX. The
next direction is a route-level audit and handoff cockpit:

```text
Evidence hub -> evidence detail -> guarded download -> archive builder ->
archive detail -> source object timeline -> artifact detail -> audit search.
```

This keeps Evidence Hub backend-first while making it useful as the system's
cross-module proof layer.
