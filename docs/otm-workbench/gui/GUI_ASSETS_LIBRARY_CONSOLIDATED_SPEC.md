## 1. Product Objective

Assets Library should be the governed library of reusable implementation files
for OTM Workbench projects.

The module exists to manage:

```text
templates
payload samples
specifications
diagrams
XML files
collections
mapping exports
schema packs
project playbooks
```

Assets Library must stay conceptually separate from:

- **Artifact**: a technical output generated by a job, validation, export, or
  package build.
- **Evidence**: an auditable proof record that something was validated,
  decided, executed, or reviewed.

An asset is a reusable governed file. It may link to artifacts or evidence, but
it is not automatically the same thing.

## 2. Current UI Review

The current `/assets` screen is a strong backend-backed functional slice. It
proves asset listing, backend-owned filters, selected asset detail,
classification authoring, create/update metadata, version upload, link
creation, guided target selection, Evidence Hub target filters, guarded
download, archive lifecycle, disabled actions, and backend validation.

Browser review found that it is not yet the final operating model:

1. `/assets` combines library browsing, create asset, classification authoring,
   version upload, link creation, download, and archive in one stateful route.
2. The current workflow stages (`Library`, `Create`, `Version`, `Link`,
   `Lifecycle`) are technically clear to the implementation, but they do not
   tell a user a natural story.
3. The `Create` stage mixes two different jobs: classification authoring and
   asset metadata authoring.
4. The `Version` stage is a focused action, but it should be attached to an
   asset detail route or a dedicated upload route, not a global stage.
5. The `Link` stage works technically, but target selection should feel like a
   relationship editor for one asset.
6. The `Lifecycle` stage contains destructive/important actions like archive,
   which should open a dedicated route with impact/confirmation.
7. The selected-object side panel is useful for summary, but it should not be
   the primary way to inspect an asset.
8. Current list rows are capped, but deeper search and route-level detail are
   needed before operational volumes grow.

The current screen works as contract evidence. To become genuinely useful, the
module needs route-level journeys: hub, asset library, asset detail, create,
edit metadata, upload version, manage links, classifications, download/history,
and archive.

## 3. Design Decision

`/assets` becomes the Assets Library hub. It should explain what assets exist,
what needs attention, and how to start common tasks. It should not host the
entire lifecycle.

The redesign separates Assets Library into route-level journeys:

1. **Assets Hub**
   Understand asset health, recent assets, missing versions, archived assets,
   and reusable schema/payload/template assets.

2. **Asset Library**
   Search, filter, and open assets.

3. **Asset Detail**
   Inspect one asset, metadata, current version, links, available actions, and
   lifecycle state.

4. **Create Asset**
   Create a reusable governed asset from backend-owned classifications.

5. **Edit Metadata**
   Update asset metadata with backend validation.

6. **Upload Version**
   Add a new file version to one asset.

7. **Asset Versions**
   Review version history and guarded downloads.

8. **Asset Links**
   Link an asset to modules, macro-objects, OTM tables, batches, checklists,
   artifacts, and evidence.

9. **Classifications**
   Manage backend-owned classification rows separately from asset creation.

10. **Lifecycle**
    Archive/restore/deprecate assets through dedicated impact screens.

## 5. Global UX Rules

### 5.1 Do Not Mix Classification Authoring With Asset Creation

Asset creation can use classification values, but creating/editing
classifications is an admin/configuration task. It should live in
`/assets/classifications`.

### 5.2 One Asset Owns Versions And Links

Version upload, version history, link creation, and lifecycle actions must be
scoped to one `assetId`.

### 5.3 Destructive Or Lifecycle Actions Need Dedicated Screens

Archive, retire, restore, or hard-delete decisions must open route-level
confirmation with backend impact and disabled reasons.

### 5.4 Back Button Required

Every drill-down route needs a visible `Back` action:

- `Back to Assets`
- `Back to Library`
- `Back to Asset`
- `Back to Classifications`

### 5.5 Backend Ownership

The frontend must not infer:

- asset lifecycle permissions;
- classification validity;
- version eligibility;
- download eligibility;
- archive eligibility;
- link target safety;
- artifact/evidence client-safety;
- OTM table validity;
- macro-object validity;
- sensitivity/visibility policy.

Those remain backend-owned through Assets Library, Catalog Core, Data
Dictionary, Evidence Hub, and platform permission contracts.

## 6. Screen: `/assets`

### Purpose

Orient the user and show library health without exposing every asset operation.

### Layout

- Header: `Assets Library`
- Primary action: `Create asset`
- Secondary actions:
  - `Open library`
  - `Manage classifications`
- Metrics:
  - total assets
  - versioned assets
  - assets missing current version
  - internal/client-safe assets
  - archived assets
  - visible rows/current filter count
- Sections:
  - `Recommended next actions`
  - `Recent assets`
  - `Recently versioned`
  - `Assets needing attention`

### Clicks

| User click | Opens | Backend load |
|---|---|---|
| `Create asset` | `/assets/new` | classifications, allowed scopes, permissions |
| `Open library` | `/assets/library` | searchable asset metadata |
| `Manage classifications` | `/assets/classifications` | classification groups |
| Recent asset row | `/assets/:assetId` | asset detail |
| Missing version row | `/assets/:assetId/versions/new` | asset + upload eligibility |

### What Must Not Be Here

- classification create form;
- full asset metadata edit form;
- file upload control;
- link target authoring;
- archive confirmation.

## 7. Screen: `/assets/library`

### Purpose

Find assets by backend-owned metadata and open asset details.

### Asset Header Search Contract

Asset search must be backend-owned. The UI renders searchable fields from
backend metadata. The MVP baseline should support:

| Field | Purpose | Example |
|---|---|---|
| asset id | technical identifier | `asset_...` |
| asset name | user-facing name | `Location Template` |
| type | classification | `TEMPLATE`, `SPEC`, `SAMPLE_PAYLOAD` |
| category | classification | `OTM_SETUP`, `INTEGRATION` |
| status | lifecycle state | `DRAFT`, `ACTIVE`, `ARCHIVED` |
| visibility | visibility policy | `INTERNAL`, `PROJECT` |
| scope | reuse scope | `GLOBAL`, `PROJECT`, `MODULE` |
| sensitivity | sensitivity policy | `INTERNAL`, `CLIENT_SAFE` |
| module id | linked module | `rates`, `master_data` |
| macro object | Catalog macro-object | `RATE_RECORD`, `LOCATION` |
| OTM table | Data Dictionary table | `LOCATION`, `RATE_GEO` |
| has current version | current version exists | `true`, `false` |
| linked target type | relationship type | `MODULE`, `OTM_TABLE`, `EVIDENCE` |
| target OTM version | supported baseline | `26A`, `26B` |
| tags | reusable qualifiers | `schema,payload,test` |
| description | searchable explanation | `Reusable integration payload sample` |

Every searchable text field supports:

| Operator | Behavior |
|---|---|
| `begins with` | case-insensitive prefix match |
| `contains` | case-insensitive substring match |
| `one of` | comma-separated list; trims spaces around values |
| `not one of` | comma-separated exclusion list; trims spaces around values |

Search behavior rules:

- Multiple filters combine with `AND`.
- Empty filters are ignored.
- Invalid operator/value combinations are blocked before backend call.
- Backend returns normalized filters and pagination metadata.

### Clicks

| User click | Opens | Backend load |
|---|---|---|
| `Apply search` | stays on `/assets/library` | filtered asset list |
| `Reset search` | stays on `/assets/library` | default asset list |
| Asset row | `/assets/:assetId` | asset detail, versions, links, actions |
| `Create asset` | `/assets/new` | classifications |
| `Upload version` row action | `/assets/:assetId/versions/new` | asset + upload eligibility |
| `Archive` row action | `/assets/:assetId/archive` | impact summary |

|---|---|---|
| `Create link` | backend link create | link appears in list |
| `Reset target filters` | route-session reset | target search clears |

### Backend-Owned Target Types

The backend owns available target types and target lookup rules:

- `MODULE`
- `MACRO_OBJECT`
- `OTM_TABLE`
- `BATCH`
- `CHECKLIST`

- `EVIDENCE`

`ARTIFACT` and `EVIDENCE` targets must be verified against client-safe Evidence

## 18. Backend Contract Alignment

The redesign should reuse existing backend contracts:

- `GET /api/v1/modules/assets/assets`
- `GET /api/v1/modules/assets/classifications`
- `POST /api/v1/modules/assets/classifications`
- `PATCH /api/v1/modules/assets/classifications/{classification_id}`
- `POST /api/v1/modules/assets/assets`
- `GET /api/v1/modules/assets/assets/{asset_id}`
- `PATCH /api/v1/modules/assets/assets/{asset_id}`
- `POST /api/v1/modules/assets/assets/{asset_id}/archive`
- `GET /api/v1/modules/assets/assets/{asset_id}/download`
- `GET /api/v1/modules/assets/assets/{asset_id}/versions`
- `POST /api/v1/modules/assets/assets/{asset_id}/versions`
- `GET /api/v1/modules/assets/assets/{asset_id}/links`
- `POST /api/v1/modules/assets/assets/{asset_id}/links`

Potential backend/API improvements:

1. searchable/paginated asset library endpoint with normalized filters;
2. route-optimized asset detail endpoint with metadata, current version, links,
   actions, and disabled reasons;
3. classification management endpoint grouped by type and protection state;
4. target lookup endpoint for link creation with source module, macro object,
   table, artifact, and evidence filters;
5. archive impact endpoint before archive commit;
6. schema pack asset subtype if WSDL/XSD support becomes first-class.

## 19. QA Journeys

Functional browser QA must cover:

1. open hub, open library, search/filter, reset search, open asset detail;
2. create asset with valid metadata;
3. invalid macro-object and invalid OTM-table metadata recovery;
4. edit selected asset metadata after switching selected assets;
5. upload version and download current version through guarded backend;
6. create links for `MODULE`, `MACRO_OBJECT`, and `OTM_TABLE`;
7. create `ARTIFACT`/`EVIDENCE` links only for client-safe Evidence Hub targets;
8. reset overly narrow Evidence Hub target filters;
9. archive asset and verify update/upload/link are blocked;
10. switch assets and verify stale upload file, link target draft, filters, and
    feedback are cleared.

## 20. Implementation Slices

### Slice 1: Hub And Library

- Keep `/assets` as hub.
- Build `/assets/library`.
- Add backend-owned search metadata, operators, pagination, and row actions.

### Slice 2: Asset Detail

- Build `/assets/:assetId`.
- Move selected-object side panel content into route-level detail.
- Add route links for edit, versions, links, download, and archive.

### Slice 3: Create And Edit Metadata

- Build `/assets/new` and `/assets/:assetId/edit`.
- Keep Catalog/Data Dictionary metadata validation backend-owned.

### Slice 4: Versions And Downloads

- Build `/assets/:assetId/versions` and `/assets/:assetId/versions/new`.
- Keep download guarded and backend-audited.

### Slice 5: Links And Target Lookup

- Build `/assets/:assetId/links`.
- Use backend-owned target search for module, macro object, OTM table, artifact,
  and evidence links.

### Slice 6: Classifications And Lifecycle

- Build classification list/create/edit routes.
- Build archive confirmation route.
- Keep system-protected and archived-state rules backend-owned.

### Slice 7: QA And Documentation

- Update React/browser QA for route leave/return and asset switching.
- Update Linear and module docs.
- Keep synthetic data only.

## 21. Acceptance Criteria

The redesign is accepted when:

1. `/assets` explains library health without hosting the full asset lifecycle.
2. Asset rows open route-level detail pages with `Back`.
3. Create, edit, version upload, links, classifications, and archive each have a
   clear route-level task boundary.
4. Asset search supports backend-owned metadata fields and operators.
5. Classification authoring is separated from asset creation.
6. Archived asset blockers and disabled reasons come from backend contracts.
7. Artifact/evidence link target safety remains backend-owned.
8. Guarded downloads do not expose local file paths.
9. Functional QA covers happy, negative, out-of-order, and route recovery paths.
10. No real client data appears in docs, UI fixtures, screenshots, or tests.

## 22. Explicit Non-Goals

- Do not turn Assets Library into Evidence Hub.
- Do not treat every generated artifact as an asset automatically.
- Do not store or expose real client data.
- Do not expose local filesystem paths.
- Do not allow frontend-only lifecycle or permission decisions.
- Do not allow global/shared visibility for sensitive assets without backend
  policy.
